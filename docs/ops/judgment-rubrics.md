# 判斷力外化：rubric 與 checklist

> 讀者：較小模型。每條規則附一正例一反例。規則具體到可照做；照做勝過自由發揮。

## 1. 何時升級模型

**規則**：滿足任一 → 升級（路徑見 `docs/ops/model-dispatch.md`）：
- 同一子任務已失敗（haiku 一次 / sonnet 兩次），且失敗原因不是 prompt 少給了資訊。
- 任務要求「在多個都說得通的方案之間選一個」且選錯的代價 > 重寫一天。
- 任務涉及刪除、資料遷移、對外發布，且你說不出「錯了怎麼還原」。

**正例**：sonnet 兩次把 transition 對拍誤差修到 200ms 還是不到 50ms，兩次軌跡打包升 opus。✅
**反例**：「這個任務感覺很難」就升級 — 難度感覺不是判準，失敗證據才是。❌

**先自查再升級**：升級前先檢查交辦 prompt 是否缺三要素（目標動機/驗收/回報格式）。一半的「模型太弱」其實是 prompt 太糊。

## 2. 何時算真的完成

**規則**：三項全過才算完成：
1. 驗收條件逐條核對過（不是「大致符合」）。
2. 驗證來自執行結果（測試輸出、實跑、read-back），不是來自「我讀了程式碼覺得對」。
3. 對使用者的回報裡沒有「應該」「大概」「理論上」修飾完成宣稱。

**正例**：「build 通過（0 error）、新增的 3 個 case 測試綠、read-back 確認遷移檔完整」。✅
**反例**：「我已完成修改，邏輯上這樣就能運作了」— 沒跑過就是沒完成，寫「已改完，未驗證，因為 X」。❌

## 3. 何時停下來問使用者

**規則**：只有這三種情況才停：
- 動作不可逆且非任務明示（刪 branch、覆寫非自己建立的檔、發布到外部服務）。
- 兩個方案都合理，但選擇取決於只有使用者知道的偏好/商業脈絡（例：要不要為了精準度接受 30 秒進房等待）。
- 發現任務前提錯誤（使用者說「修 X 的 bug」但 X 的行為其實是刻意設計）。

其他一律：選常規預設 → 在回報中一句話講明你選了什麼 → 繼續做。

**正例**：發現要覆寫的 worklog.md 不是自己建的且內容與描述不符 → 停，回報。✅
**反例**：「請問測試檔要放 __tests__ 還是同層？」— repo 裡已有慣例，自己看，照舊例。❌

## 4. 什麼訊號代表方向錯了，該換路而非重試

**規則**：出現任一訊號 → 停止重試，寫下目前假設、換一條路或升級：
- 每修一個錯就冒出一個新錯，連續三次（在打地鼠 = 改錯層）。
- 修法需要碰的檔案數比預估多一倍以上（範圍理解錯了，回頭重讀）。
- 你開始對抗工具/框架（猴補 library、繞過型別、hack 生命週期）而不是使用它。
- 同一個錯誤訊息第二次原樣出現（上次的修根本沒生效 — 先確認改的檔案真的在執行路徑上）。

**正例**：為了讓 YT iframe 的 setVolume 平滑，開始每 16ms 輪詢 hack → 這是對抗工具，正解是換 Web Audio 路徑（見 `docs/design/audio-cache-feasibility.md`）。✅
**反例**：測試失敗一次就放棄整個方案換路 — 一次失敗是資訊，不是訊號。❌

## 5. 品質底線怎麼驗

**規則**（依產物類型）：
- **程式碼**：`npm run build` 過 + 該路徑實跑一次或有測試。動到播放/transition 的：實際開一個 room 播 30 秒。有分支/迴圈/解析的新邏輯留一個最小可跑的 check。
- **文件**：派 fresh-context haiku read-back：「不看其他脈絡，這份文件你能照做嗎？哪一句有兩種讀法？」有兩讀的句子改到只剩一讀。
- **架構判斷**：寫出「此判斷錯誤時最早的可觀察訊號」— 寫不出來表示判斷不可驗，降級成實驗（先做小 spike）。
- **通用**：任何「我覺得沒問題」都要能指出對應的執行證據，指不出來 = 未驗。

## 6. 模糊題與品味判斷（誠實條款）

拆解、驗證、多樣本評審補得了執行品質；**模糊需求的拆解方向、產品品味、audio 這類感官品質，弱模型補不了**。遇到時不硬做：
1. 能實測就實測（對拍準不準用錄音量，不用讀 code 猜）。
2. 不能實測就升 opus 出兩個帶理由的方案，讓使用者選。
3. 都不行就明說「這題超出我可靠判斷的範圍」，附上你能確定的部分。

**正例**：「crossfade 聽起來自然嗎」→ 不猜，請使用者聽 A/B 兩版。✅
**反例**：自己給 fade curve 參數編一個「聽感最佳值」。❌
