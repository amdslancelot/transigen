# 給未來 session 的信

> 寫於 2026-07-06，Fable 5 唯一一次 session。之後接手的是你 — 大概是 Sonnet 或 Opus。
> 這封信講三件使用者沒問但重要的事、這套制度會怎麼壞、以及我對哪些產出信心最低。

## 三件使用者沒問但我認為最重要的事

**1. 這個專案最缺的不是設計，是「可量測的驗收」。**
repo 沒有任何測試，而產品核心（transition 對不對拍）是感官品質 — 弱模型最補不了的那種。最高槓桿的一小步：做一個量測工具（錄下 transition 輸出、離線算切點與 beat grid 的誤差 ms 數），讓「對拍準不準」從品味題變成數字題。有了數字，Sonnet 就能安全地迭代 transition 邏輯；沒有數字，每次改動都在賭。這比實作任何新功能都優先。

**2. 動工 audio 架構前，先花一小時做實驗，不要先寫程式。**
`docs/design/audio-cache-feasibility.md` 的最大不確定性是 yt-dlp 在 server IP 上的實際成功率（這是持續變動的軍備競賽，我的數字寫下當天就在過期）。正確的第一步：租最便宜的 VPS，跑 yt-dlp 下載 20 首測試歌，記錄成功率與速度。結果直接決定 proxy 預算與 fallback 設計的比重。一小時的實驗勝過三天的設計討論。

**3. 使用者的額度哲學：主對話短命、檔案長命。**
使用者明確把模型額度當稀缺預算管理。你的每個結論如果只活在對話裡，session 一斷就得重新花錢想一次。所以：中間結論隨做隨寫進檔案；跨 session 事實進記憶目錄；worklog.md 是使用者自己的習慣，尊重它、別重構它。另外 caveman + ponytail 兩個 plugin 每個 session 常駐注入約 2k token — 如果使用者哪天抱怨額度，這是可以建議關掉的成本（要先問，見 maintenance.md 權限分級）。

## 這套制度最可能的退化方式與預防

1. **索引腐化 → 信任崩壞（最可能）**：CLAUDE.md 地圖裡的路徑改名失效 → 弱模型發現索引不可信 → 回到整包掃 repo 的老路，制度形同虛設。預防：發現失效路徑當場修（權限已開放，見 maintenance.md）；健檢清單第 2 條有一行 shell 驗證法。
2. **規則在壓力下被跳過**：任務急的時候，弱模型會「這次先自己來」不派工、不驗證。一次沒事，三次成習慣。預防：judgment-rubrics.md#2 把「未驗證」排除在「完成」定義外 — 只要完成定義守住，跳過的步驟會在回報時暴露。
3. **lessons.md 變垃圾場**：什麼都往裡丟，超長之後沒人讀。預防：只有「重試兩輪才解掉」等級的事件才進 lessons（門檻寫在 maintenance.md），超 30 條強制精簡。
4. **制度自我膨脹**：未來 session 覺得「再加一條規則就好了」，ops 檔越長越沒人讀。預防：200 行上限 + 改判準要問使用者。刪規則的價值通常高於加規則。

## 我對哪些產出信心最低（誠實排序）

1. **feasibility 文件裡的所有數字**（下載秒數、yt-dlp 節流行為、±20–30ms 的 getCurrentTime 回歸精度）：方向對，數字是量級估計，沒實測。特別是 getCurrentTime 對齊精度那條 — 動工 Phase 1 時第一件事就是量它，量出來不到 50ms 就照做，超過就把 Phase 1 的預期降為「粗對拍」。
2. **模型/參數清單**（model-dispatch.md）：alias 與 effort 欄位是官方文件查證的，但查證 agent 的回報裡有明顯的雜訊（把 alias 誤標成 "Claude 3 variants"），我只保留了交叉確認過的部分。型號 ID 來自本 session 的 harness 環境，隨時會過期 — 用之前看查證日期。
3. **雙路徑抽象的建議**（feasibility 風險 4）：對這個規模的 codebase，「同一介面包 iframe 與 Web Audio」可能是過度設計。實作時如果發現介面抽象比兩套 if/else 還長，就用 if/else，別為了我這句話硬抽象。
4. **Fable→Opus 導流的額度問題**：完全未確認，文件無記載。使用者要答案只能去 usage 儀表板實測。

## 最後

制度是拿來省判斷的，不是拿來崇拜的。哪條規則連續三次礙事，就帶著證據去問使用者要不要改 — 這個動作本身就寫在制度裡（maintenance.md 健檢第 4 條）。
